# Dimension: DimCustomer

## Purpose

The `DimCustomer` table stores customer-related information used for sales analysis and reporting.

It provides descriptive attributes that allow business users to analyze sales by customer location and customer identity without accessing the operational database.

---

# Source

**Operational Table**

- dbo.Customers

---

# Grain

**One record represents one customer.**

---

# Primary Key

| Column | Type | Description |
|---------|------|-------------|
| CustomerKey | INT (IDENTITY) | Surrogate key generated by the Data Warehouse |

---

# Business Key

| Column | Source |
|---------|--------|
| CustomerID | dbo.Customers.customer_id |

The business key is retained to maintain traceability back to the operational database.

---

# Attributes

| Column | Data Type | Description |
|---------|-----------|-------------|
| CustomerKey | INT | Warehouse surrogate key |
| CustomerID | VARCHAR(50) | Customer identifier from the operational database |
| CustomerUniqueID | VARCHAR(50) | Unique customer identifier |
| ZipCodePrefix | INT | Customer ZIP code prefix |
| CustomerCity | VARCHAR(100) | Customer city |
| CustomerState | CHAR(2) | Customer state |

---

# Data Source Mapping

| Data Warehouse | Operational Database |
|----------------|----------------------|
| CustomerID | customer_id |
| CustomerUniqueID | customer_unique_id |
| ZipCodePrefix | customer_zip_code_prefix |
| CustomerCity | customer_city |
| CustomerState | customer_state |

---

# ETL Process

```
dbo.Customers
        │
        ▼
Extract
        │
        ▼
Transform
        │
        ▼
Load
        │
        ▼
dw.DimCustomer
```

The ETL process copies customer records from the operational database into the Data Warehouse while generating a surrogate key (`CustomerKey`) for each customer.

---

# Relationships

The `DimCustomer` table is related to the `FactSales` table through the `CustomerKey` surrogate key.

```
DimCustomer
      │
      │ CustomerKey
      ▼
FactSales
```

---

# Business Usage

This dimension enables reporting such as:

- Sales by Customer
- Sales by City
- Sales by State
- Customer Distribution
- Revenue by Geographic Location

---

# Design Decisions

- Uses a surrogate key (`CustomerKey`) instead of the operational `CustomerID`.
- Retains the original `CustomerID` as the business key.
- Stores descriptive customer attributes only.
- Does not contain transactional or sales measures.
- Designed according to Kimball Dimensional Modeling principles.

---

# Future Enhancements

The dimension can be extended to support Slowly Changing Dimensions (SCD Type 2) by adding columns such as:

- EffectiveDate
- ExpiryDate
- IsCurrent

This will allow historical tracking of customer attribute changes without redesigning the warehouse.
