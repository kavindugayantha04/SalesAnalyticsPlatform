# Dimension: DimPayment

## Purpose

The `DimPayment` table stores payment method information used for sales analysis and business reporting.

It provides descriptive information about payment methods, enabling users to analyze customer purchasing behavior and payment preferences without querying the operational database.

---

# Source

**Operational Table**

- dbo.Payments

---

# Grain

**One record represents one payment type.**

---

# Primary Key

| Column | Type | Description |
|---------|------|-------------|
| PaymentKey | INT (IDENTITY) | Surrogate key generated by the Data Warehouse |

---

# Business Key

| Column | Source |
|---------|--------|
| PaymentType | dbo.Payments.payment_type |

The business key is retained to maintain traceability back to the operational database.

---

# Attributes

| Column | Data Type | Description |
|---------|-----------|-------------|
| PaymentKey | INT | Warehouse surrogate key |
| PaymentType | VARCHAR(50) | Payment method used by the customer |

---

# Data Source Mapping

| Data Warehouse | Operational Database |
|----------------|----------------------|
| PaymentType | payment_type |

---

# ETL Process

```text
dbo.Payments
      │
      ▼
Extract
      │
      ▼
Remove Duplicate Payment Types
      │
      ▼
Load
      │
      ▼
dw.DimPayment
```

The ETL process extracts all payment methods from the operational database, removes duplicate values using the `DISTINCT` keyword, and loads the unique payment methods into the Data Warehouse.

---

# Relationships

The `DimPayment` table is related to the `FactSales` table through the `PaymentKey` surrogate key.

```text
DimPayment
      │
      │ PaymentKey
      ▼
FactSales
```

---

# Business Usage

This dimension enables reporting such as:

- Sales by Payment Method
- Payment Method Distribution
- Revenue by Payment Type
- Customer Payment Preferences
- Payment Method Trend Analysis

---

# Design Decisions

- Uses a surrogate key (`PaymentKey`) instead of the payment method name.
- Stores unique payment methods only.
- Removes duplicate payment types during the ETL process.
- Contains descriptive information only and no transactional measures.
- Designed according to Kimball Dimensional Modeling principles.

---

# Future Enhancements

Additional payment-related attributes can be added if available, such as:

- Payment Category
- Digital/Offline Payment Indicator
- Installment Type
- Payment Provider

This allows the dimension to grow without affecting the existing warehouse design.
