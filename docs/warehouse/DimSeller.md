# Dimension: DimSeller

## Purpose

The `DimSeller` table stores seller-related information used for sales analysis and business reporting.

It provides descriptive attributes that allow users to analyze sales performance by seller and geographical location without querying the operational database.

---

# Source

**Operational Table**

- dbo.Sellers

---

# Grain

**One record represents one seller.**

---

# Primary Key

| Column | Type | Description |
|---------|------|-------------|
| SellerKey | INT (IDENTITY) | Surrogate key generated by the Data Warehouse |

---

# Business Key

| Column | Source |
|---------|--------|
| SellerID | dbo.Sellers.seller_id |

The business key is retained to maintain traceability back to the operational database.

---

# Attributes

| Column | Data Type | Description |
|---------|-----------|-------------|
| SellerKey | INT | Warehouse surrogate key |
| SellerID | VARCHAR(50) | Seller identifier from the operational database |
| ZipCodePrefix | INT | Seller ZIP code prefix |
| SellerCity | VARCHAR(100) | Seller city |
| SellerState | CHAR(2) | Seller state |

---

# Data Source Mapping

| Data Warehouse | Operational Database |
|----------------|----------------------|
| SellerID | seller_id |
| ZipCodePrefix | seller_zip_code_prefix |
| SellerCity | seller_city |
| SellerState | seller_state |

---

# ETL Process

```text
dbo.Sellers
      │
      ▼
Extract
      │
      ▼
Transform
      │
      ▼
Load
      │
      ▼
dw.DimSeller
```

The ETL process copies seller records from the operational database into the Data Warehouse while generating a surrogate key (`SellerKey`) for each seller.

---

# Relationships

The `DimSeller` table is related to the `FactSales` table through the `SellerKey` surrogate key.

```text
DimSeller
     │
     │ SellerKey
     ▼
FactSales
```

---

# Business Usage

This dimension enables reporting such as:

- Sales by Seller
- Top Performing Sellers
- Sales by Seller City
- Sales by Seller State
- Geographic Distribution of Sellers

---

# Design Decisions

- Uses a surrogate key (`SellerKey`) instead of the operational `SellerID`.
- Retains the original `SellerID` as the business key.
- Stores descriptive seller attributes only.
- Does not contain transactional or sales measures.
- Designed according to Kimball Dimensional Modeling principles.

---

# Future Enhancements

The dimension can be extended to support Slowly Changing Dimensions (SCD Type 2) by adding columns such as:

- EffectiveDate
- ExpiryDate
- IsCurrent

This will allow historical tracking of seller information if business requirements change in the future.
